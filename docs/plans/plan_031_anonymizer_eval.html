Plan 031 — Anonymizer model/prompt eval (turn vibes into numbers)

Goal. A small, hand-run harness that measures how well a given model + prompt anonymizes a statement — PII recall (leaks), over-scrub rate (fidelity cost), and latency — over a labeled synthetic set, so "which model / which prompt" becomes a decision with evidence instead of a vibe. Mirrors scripts/eval_categorization.py / plan_020, applied to plan 030's anonymizer.

Why this exists (Exhibit A: gemma4:12b). It was picked from paper specs (non-thinking, multilingual, benchmark-friendly) and failed empirically — never terminated under Ollama's structured JSON, stalled the queue, wasted ~40 min, and even filled the disk. A one-shot eval would have flagged it in seconds. Model choice here must be data-driven; this makes that cheap and repeatable.

What it measures — and why the metric is the lesson

Anonymization has an asymmetric cost structure, and the harness is built around it:

MetricMeaningWeight
PII recallof the labeled personal-data tokens, how many did the pipeline replace? A miss = a real leak.dominant — this is the number
Over-scrub rateof non-PII text tokens, how many were replaced? Costs fidelity. secondary — over-scrubbing is safe
Latencywall time per document.practicality axis
Clean / erroreddid the model terminate and return valid JSON at all? a hard gate — a non-terminating model (gemma) fails here first

The takeaway worth internalizing: the metric must match the task's cost structure. Here a missed name is catastrophic and an over-scrubbed label is harmless, so recall >> precision — the opposite weighting you'd use for spam filtering. Building the harness forces you to feel that.

D1 — isolate the MODEL's contribution

The deterministic layer (amounts, dates, NIF, cards, postal codes, denylist identity) is scrubbed regardless of model, so counting it would wash out model differences. Recommendation: label and score recall on the model-dependent PII only — free-text names, street words, localities the rules can't shape-match. That number is what actually moves when you swap models or edit the prompt.

The eval set

A handful of synthetic statements (receipt-style + bank-style) with realistic-but- obviously-fake names/addresses, plus a labels file marking which tokens are model-dependent PII (the ground truth). Recall/precision are then plain set arithmetic against the labels — no LLM-as-judge, no subjectivity.

D2 — the eval set is COMMITTED (unlike the categorization one)

eval_categorization.py's labels are your real descriptions, so they're gitignored. Here the eval docs are synthetic and obviously-fake, so they can (and should) be committed — reproducible, shareable, and a regression guard for prompt changes. Recommendation: commit them under tests/fixtures/anon_eval/ (PDF + labels), authored the usual way (hand-authored fake table, my sign-off). Never a real statement.

How it compares models

uv run python scripts/eval_anonymizer.py --models qwen2.5:3b qwen2.5:14b

model         PII recall   over-scrub   latency/doc   status
qwen2.5:3b       7/11        4%            34s          ok
qwen2.5:14b     10/11        6%           2m10s         ok
gemma4:12b        -           -             -          ERROR (no termination)

Loop the candidate models, run each over the eval set through the real classifier, tabulate. Same "hits Ollama, so NOT in pytest" stance as eval_categorization — it's an operator tool.

What it reuses / respects

  • Pattern + conventions from scripts/eval_categorization.py (standalone script, config- driven, by-hand, reports a task-specific failure number).
  • The anonymizer's own Classifier + detect_value_type + force-replace path — it scores the real pipeline, not a reimplementation.
  • Offline suite stays offline: the script is not imported by pytest; it hits the live model.
  • No real values anywhere — synthetic fixtures only.
D3 — scope: a tool, not a framework

Recommendation: ~5 eval docs, three numbers, one table. Resist building a generic eval framework. If it grows tentacles (weighting knobs, plugins, HTML reports), it's over-built — the value is the discipline, not the software.

Double duty: prompt A/B + regression guard

The same harness A/B-tests a prompt change on a fixed model (did recall go up without over-scrub exploding?) — exactly how eval_categorization drives down junk-drawer FPs. So it also guards against a future prompt edit silently regressing name-catching.

Slice

  1. Synthetic eval fixtures + labels (hand-authored fake, signed off).
  2. scripts/eval_anonymizer.py: load set, run each model, compute recall/over-scrub/latency, print the table + per-doc misses (which PII tokens leaked).
  3. README note under the anonymize section + a short mention in plan_030.

Non-goals

  • Not a general model benchmark — it answers "which model/prompt for this task."
  • No LLM-as-judge (the labels are ground truth; keep it deterministic).
  • Not wired into the pipeline or pytest.

Open questions

  • How many eval docs is "enough" to be meaningful without being a chore? (start ~5, grow if noisy)
  • Do we score third-party counterparty names separately from the account-holder pattern? (they're the hardest recall case — maybe a column of their own)
