Plan 029 — Self-generating parser (autonomous parser-dev harness)

Goal. When a statement arrives from an institution cruzar has no parser for, an operator-run tool detects the gap, produces a privacy-safe anonymized sample (plan 030), dispatches a Claude Code harness (plan → review → implement → verify) that drafts a new parser + fixture against that sample, and opens a PR for me to review. I stay the merge gate; nothing lands unattended.

This introduces new ADRs and sits OUTSIDE the money path. An autonomous parser-dev subsystem is a decision, not a detail — approve the ADRs before we build. And it is operator-run tooling beside cruzar, never wired into pipeline.process. Wiring it into the pipeline would break "the offline suite must STAY offline" and turn a statement ingest into a network/agent-spawning event. The pipeline's only job here is to emit a clean signal; the harness is a separate command I run.

The trigger (grounded in current code)

Two distinct signals already exist in ingest_inbox (pipeline.py). We build on the intentional one:

SituationCurrent outcomeUse as trigger?
PDF in a folder with no configured accountunresolved_account No — ambiguous (could be a misfiled file)
Account declared with an institution not in PARSERS get_parser raises ValueError("no parser registered …") → caught as parse_failedYes — an explicit "this is BankX, build it" signal
Known parser, degraded layoutExtractionFallback → LLM extractor No — that's a different, existing path
D1 — a distinct no_parser status

Today "no parser registered" is swallowed into parse_failed alongside genuine parser bugs. The harness must never confuse "I haven't built BankX yet" (go draft one) with "the ActivoBank parser threw on real input" (that's a loud bug to fix, not to autogenerate around). Recommendation: distinguish the missing-parser case into its own processed_files.status = 'no_parser'. This is a schema/behavior change (guarded migration per the schema invariant) and touches pipeline flow — flagging, not deciding.

Architecture — the operator loop

A thin "shell" driven by the local model. It routes and reports; it does not reason about parsing (Claude does that). Stages:

  1. Detect. After a cruzar process run, scan processed_files for no_parser rows. For each, resolve the offending source PDF in gitignored /data/ and the declared institution.
  2. Anonymize (plan 030). Produce a structurally-faithful, value-scrubbed sample. Only this output is allowed to cross to Claude. If anonymization fails its safety gate, abort this institution — never fall back to the raw document.
  3. Dispatch the harness. Spawn Claude Code agents on a fresh branch/worktree: plan → review the plan → implement → verify. Inputs: the anonymized sample, SPEC.md + the ADR invariants as the rubric, and the parser contract (parse(pdf_path) -> ParsedStatement, deterministic top-to-bottom order, register in parsers/__init__.py).
  4. Verify. The harness must get its own AC/parser fixture test green plus ruff + pyright + full suite (the repo's existing "done" bar).
  5. Open a PR. One institution, one branch, one PR, no auto-merge. The PR body surfaces the proposed obviously-fake transaction table for my oracle sign-off (see below).

The human-in-the-loop boundary (the crux)

The harness produces a reviewable draft, never a merged parser. CLAUDE.md forbids self-generating the oracle: a parser that agrees with a fixture it also wrote proves nothing. So the harness may check internal invariants (Decimal not float, debits negative, ordering, dedup, ruff/pyright) but the committed fixture stays hand-authored, obviously-fake, and signed off by me — that is a standing procedure, not a decision this plan makes. The PR review is the oracle step: the PR presents "here is what I extracted; confirm these numbers match your real statement," and I approve or send back.

Bounded autonomy

  • One no_parser institution per invocation → one branch → one PR. No batching, no retries into main.
  • Runs only when I invoke it — never auto-fired by process.
  • The harness works in an isolated worktree; the operator loop never touches the real DB or /reports/.
  • Documents only — no images/OCR (explicitly out of scope for now).

What the verify agent checks (rubric from existing ADRs)

  • Money is Decimal, parsed straight from strings; no float anywhere.
  • Amounts native-currency, signed (debits negative); locale comma-decimals normalized at the parser boundary only.
  • Deterministic top-to-bottom emission; no merging/coalescing of distinct lines; wrapped descriptions reassembled, not merged.
  • Registered in parsers/__init__.py; implements the parse contract; one module per institution.
  • ruff clean, pyright strict clean, full suite green.

Slices

  1. Slice 1 — Anonymizer (plan 030). Foundational and independently useful; the riskiest privacy component. Build and trust it first. Nothing crosses to Claude until it's solid.
  2. Slice 2 — Detect + dispatch. no_parser status (D1), the operator loop, and harness dispatch on an anonymized sample into a worktree.
  3. Slice 3 — PR + review packaging. Branch/PR creation, the oracle-sign-off PR body, no-auto-merge guardrails.
D2 — where the code lives

This is dev/meta tooling, not part of the cruzar ingest package. Recommendation: a separate top-level location (e.g. tools/parsergen/ or scripts/) so it never imports into the money path and never ships in the pipeline. Keeps "no new top-level modules without a reason" satisfied — the reason is isolation from the pipeline.

D3 — privacy posture

Even anonymized, this sends a derived financial document off-machine to Claude. For a "local, privacy-preserving" app that's a posture change. Recommendation: explicit per-run opt-in, never automatic; the anonymized sample is the only artifact allowed to leave, enforced by plan 030's safety gate.

Risks

  • Plausible-but-wrong parser. Parsers fail silently (sign flips, comma-decimal mishandling, wrapped-line merges). The draft will look right and pass its own fixture. My PR review is load-bearing — the PR must make the tricky cases easy to check.
  • Anonymized sample must faithfully represent the real format (plan 030) or the parser is built against a phantom layout and fails on real input.
  • Scope creep into the pipeline. Guard against anyone wiring detection into process; it stays an operator command.

Out of scope

  • Images / scanned statements / OCR (documents with a text layer only, for now).
  • Auto-merge or unattended landing of any parser.
  • Non-parser institutions (fetchers, APIs).
