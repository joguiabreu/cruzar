Plan 012 — LLM categorization (Section 5 · AC4b/15/16/17/18)

Status: draft for review. Adds the llm tier to categorization (ADR-13): for transactions no rule matched, ask the local LLM to propose a merchant + category, persist the proposal (ADR-12, never recomputed), auto-apply confident in-vocabulary proposals, and surface the rest in the report's conditional Section 5 — Needs Categorization.

Today: categorize.py is rule-only; none rows never get an LLM proposal; Section 5 doesn't exist; no LLM dependency is installed.

The contract (SPEC + ADRs)

  • ADR-13 authority: manual > rule > llm. A rule may overwrite an llm row; nothing overwrites manual; none is eligible for an LLM proposal.
  • ADR-2: LLM returns only schema-constrained JSON (no free text, no math — ADR-1).
  • ADR-12: proposals persisted and never recomputed — a re-run over unchanged data makes zero LLM calls (AC15).
  • Section 5 (conditional): shown iff un-categorized merchants exist. Columns: Raw Description | LLM-Proposed Merchant | LLM-Proposed Category.
  • Edge cases (SPEC): low-confidence → Needs Categorization, no auto-assign; off-vocabulary category → treat as un-categorized, never silently create a category.

Scope (read first — D1)

DECIDE D1 — This slice does categorization only; defer the extraction fallback. AC4 has two LLM uses: (a) extraction when pdfplumber returns <50% of expected columns, and (b) categorization of unmatched merchants. (a) is a whole separate machine (LLM reading PDFs) — I propose deferring it to its own slice and implementing (b) here. AC16 explicitly accepts "trips the <50%-columns fallback or an unmatched transaction", so the categorization path satisfies AC15/16/17/18 and the (b) half of AC4. The AC4 acceptance test this slice adds covers clause (b); clause (a) stays unimplemented and noted. Confirm scope split, and that leaving AC4(a) for later is OK.

Open decisions — please annotate

DECIDE D2 — LLM client library. ADR-2 names instructor/outlines. Recommend instructor + an OpenAI-compatible client pointed at Ollama (http://localhost:11434/v1): a Pydantic model gives schema-constrained JSON with the least glue, and instructor handles retries/validation. Adds deps instructor, openai, pydantic. The client sits behind an injectable Protocol (below) so the offline suite never imports or calls it. Confirm instructor, or prefer raw ollama / outlines?
DECIDE D3 — Persistence model = new llm_categorizations cache table, keyed by exact description_raw. One LLM call per distinct raw description; repeats reuse it; re-runs read it → zero calls (ADR-12). Exact key (not normalized) for v1 — simple and safe; near-duplicate descriptions each cost one call (noted as a known cost). Columns: 
llm_categorizations(
  description_raw TEXT PRIMARY KEY,
  proposed_merchant  TEXT NOT NULL,
  proposed_category  TEXT NOT NULL,
  confidence         REAL NOT NULL,
  status             TEXT NOT NULL CHECK (status IN ('applied','needs_review')),
  model              TEXT NOT NULL,     -- which model produced it (for a future --reextract)
  created_at         TEXT NOT NULL
)
New table via CREATE TABLE IF NOT EXISTS in schema.sql — this is the migration path for a new table (init runs it on old DBs too); schema-parity holds with no _migrate step (that rule is for altering existing tables). Confirm table + exact-description key.
DECIDE D4 — Auto-apply = upsert a merchant row, link the txn as llm. When a proposal is confident and its category is in-vocabulary (status='applied'): upsert a merchants row (name=proposed_merchant, category=proposed_category) and set transactions.merchant_id + merchant_source='llm'. Keeps Spending/Earning Detail uniform (one merchant_id join) and matches the data model (merchant_source='llm' alongside the FK). A later rule still overrides it (ADR-13). The alternative — text-only proposal, NULL merchant_id, special-cased in report queries — is messier. Confirm upsert-merchant approach.
DECIDE D5 — Confidence threshold + config. Proposal carries a confidence ∈ [0,1]; < min_confidence → needs_review (Section 5, not applied). Add an llm: block to cruzar.yaml: 
llm:
  enabled: true                 # false → rule-only, no calls (offline users)
  model: qwen3:8b               # was top-level llm_model; kept back-compatible
  host: http://localhost:11434
  min_confidence: 0.7
Confirm the 0.7 default and the llm: block shape.
DECIDE D6 — Degradation when Ollama is unavailable. Categorization is enrichment, not source parsing — so per the CLAUDE.md integration rule it should degrade, not crash: wrap transport/parse failures into an LlmError, log a warning, leave those rows none (they appear in Section 5), and finish the run. Confirm degrade-don't-crash (the report still renders without a running LLM).
DECIDE D7 — Section 5 scope = this report-month's none cash transactions. Consistent with Sections 2/3 ("this month"). Each distinct raw description shows once with its persisted proposal (the needs_review one if present, else blank). Confirm month-scoped (vs all-time un-categorized).
DECIDE D8 — Defer recategorize + --reextract CLI. ADR-13's cruzar recategorize (manual tier) and ADR-12's --reextract are real but separable; this slice ships automatic llm proposals + caching + Section 5. AC17 (manual frozen) is testable by seeding a manual row directly. Confirm deferral.

Settled (derived — stated, not asked)

SETTLED Off-vocabulary proposed category → needs_review, never create a category (SPEC edge).
SETTLED Re-run flow (ADR-12/13): rules run first; a rule matching an llm row demotes it to rule with no call (AC18); none rows are looked up in the cache before any call; an already-llm row is left as-is (no call). Net: zero calls on an unchanged re-run (AC15).
SETTLED Computations stay in Python/SQL (ADR-1); the LLM only proposes {merchant, category, confidence}.

Design

The injectable client (offline-safe)

# categorize.py
class Proposal(NamedTuple):
    merchant: str
    category: str
    confidence: float

class LlmCategorizer(Protocol):
    def propose(self, description: str, categories: list[str]) -> Proposal | None: ...
        # returns None on a clean "I don't know"; raises LlmError on transport failure

def categorize(conn, *, propose: LlmCategorizer | None = None,
               model: str = "", min_confidence: float = 0.7) -> None:
    # 1. rule pass (existing): manual frozen; rule sets/overrides; else -> none
    # 2. llm pass over remaining 'none' rows, grouped by description_raw:
    #    - cache hit -> apply or skip per stored status; NO call
    #    - cache miss and propose is None -> leave 'none' (rule-only / disabled)
    #    - else call propose(); classify -> upsert llm_categorizations; apply if 'applied'
  • Tests inject a fake propose (counting spy / raise-on-call) — the real Ollama client lives in a new llm.py and is only built by pipeline.process when llm.enabled.
  • "applied" iff confidence ≥ min_confidence AND proposed_category ∈ categories.

llm.py (new — the only module that imports instructor/openai)

  • ollama_categorizer(model, host) -> LlmCategorizer: builds the instructor client; propose() sends the description + the controlled category list, parses a Pydantic Proposal, wraps any transport/validation failure as LlmError.
  • Prompt: "Given this bank-statement description, name the merchant and pick exactly one category from this list; give a 0–1 confidence." No amounts, no math (ADR-1).

report.py — Section 5

  • _needs_categorization_section: the report-month's cash transactions with merchant_source='none'; one row per distinct description with its needs_review proposal (merchant/category) if persisted, else blank. Rendered only if non-empty (AC9). Appended after Investment Detail.

pipeline.py / config.py / cruzar.yaml

  • config: parse the llm: block (back-compat: fall back to top-level llm_model).
  • pipeline.process: build the categorizer when enabled (else None); pass propose + min_confidence to categorize.categorize.
