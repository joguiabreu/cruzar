Plan 027 — Manual is-transfer override

Slice goal. Let me say "this is a transfer" or "this is not a transfer" by hand, and have that decision survive the per-run transfer recompute. This is Phase 2 of the manual-override roadmap (after plan 026). It closes the gap the IPS counterparty-transfer case exposed: a line wrongly paired as a transfer (so excluded from Spent) — or a real transfer the detector missed — currently can't be corrected.

This touches transfer detection — stop-and-ask territory. transfers.py deliberately has no manual override today: is_transfer is fully recomputed every run (rule patterns + pairing), unlike merchant_source. Adding a manual override changes that contract, so D1/D2 below need explicit sign-off, and the SPEC's transfer section + relevant ADR get a note. is_transfer drives Earned/Spent exclusion (AC19/AC21), so this must not silently shift those aggregates.

Why category alone (plan 026) isn't enough

A transfer is excluded from Earned/Spent entirely, so labeling a transfer's category "Dining" (plan 026) would NOT make it appear as Dining spending — it's still excluded. To turn a mis-paired transfer into real spending you must first flip is_transfer = 0. So the full one-off fix is often "un-transfer it (027) and set its category (026)" — the two plans compose.

Decisions needing your sign-off

D1 — a protected manual flag the recompute respects

Add a manual override that the transfer recompute treats as authoritative: a row with a manual is_transfer decision is neither re-paired nor pattern-flipped. Recommended mechanism: a nullable transactions.is_transfer_manual (1/0/NULL) — NULL means "auto" (today's behavior); set means "honor this". The recompute writes the effective is_transfer = COALESCE(is_transfer_manual, <computed>). Guarded migration; default NULL keeps every existing row identical.

D2 — pairing must not "use up" a manually-forced leg incorrectly

Pairing matches opposite-signed legs (AC21). If I force one leg's is_transfer, decide how the recompute treats its partner: a forced transfer leg should still let its true partner pair normally; a forced non-transfer leg must be excluded from pairing so it isn't re-flipped. Spell out the interaction so AC21 stays green and no aggregate moves unexpectedly.

D3 — CLI surface, consistent with 026

Extend the plan-026 command rather than add a new one: cruzar categorize <id...> --as-transfer | --not-transfer | --auto (--auto clears the override). Keeps one verb for "fix this line".

Design / modules

  • schema.sql + db._migrate — nullable is_transfer_manual (guarded migration; parity test green).
  • transfers.py — recompute honors the manual flag: exclude forced rows from re-decision, set effective is_transfer via COALESCE, keep pairing correct (D2).
  • cli.py — --as-transfer/--not-transfer/--auto on categorize.

Tests

  • Force --not-transfer on a mis-paired leg → it re-enters Spent; its partner is handled per D2; a re-run keeps the manual decision (not recomputed).
  • Force --as-transfer on a missed transfer → excluded from Earned/Spent.
  • --auto restores recompute.
  • AC19/AC21 fixtures stay green; schema-parity + smoke green.

SPEC / ADR-AC flags

  • Changes the transfer-detection contract (recompute is no longer unconditional) — SPEC transfer section + the transfer ADR get an amendment; confirm before building.
  • Guards AC19/AC21 — tests must show aggregates only move where I explicitly overrode.
  • Schema migration (guarded _migrate + parity).

Out of scope

  • Auto-learning a transfer pattern from a manual decision (that's a rule edit in flows.yaml, done by hand).

Definition of done

  • AC19/AC21 + schema-parity + smoke green; ruff/pyright/pytest clean; SPEC updated.
