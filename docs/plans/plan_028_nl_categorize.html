Plan 028 — Natural-language categorize (optional sugar)

Slice goal. Let me say "classify 122344 as Fuel" or "mark 12345 and 67890 as not transfers" in English, instead of remembering the exact flags. This is Phase 3 — pure ergonomics over the deterministic commands in plans 026/027. It is explicitly optional: ship only if the CLI proves tedious in daily use.

The model must NOT write — it only drafts. cruzar ask is read-only (ADR-17); mutations stay deterministic (ADR-1). The safe shape: the planner maps English to a structured CategorizeCommand{ids, category?, transfer?} (same constrained-output trick as the query planner), the command is printed for confirmation, and only on an explicit "yes" does the deterministic executor from 026/027 apply it. Model proposes → you confirm → Python writes. This is the only way NL is acceptable here.

D0 — do we even want this? (decide first)

For exact ids + a closed category vocabulary, typing cruzar categorize 123 456 --category Dining is faster and safer than NL, and a 3B model mis-reading an id is a real risk even with confirmation. Recommendation: defer — build 026/027, live with the CLI, and only do 028 if the friction is real. This plan exists so the decision and the safe design are on record, not to greenlight it.

D1 — separate write-mode, not inside `ask`

Keep ask a pure reader. Put NL categorize behind its own verb (e.g. cruzar categorize --say "mark 123 as Fuel") so the read-only port stays intact and the write surface is explicit. ADR-17 is untouched (no write ever flows through ask).

D2 — mandatory confirmation, echoing exact ids + resolved category

The drafted command is shown as a concrete preview — "Set txn 122344 (€xx.xx, 2026-05-10, '…') → Fuel. Apply? [y/N]" — so a mis-parsed id or category is caught by eye before any write. Non-interactive use requires --yes. Never auto-apply a model-drafted mutation.

Design / modules (if built)

  • analytics.py (or a sibling) — a CategorizeCommand pydantic model = the constrained planner output; reuse the Ollama planner pattern from llm.ollama_query_planner.
  • executor — calls the deterministic set_manual / is_transfer functions from plans 026/027 after confirmation. No new write logic.
  • cli.py — categorize --say "…" + --yes.

Tests

  • Fake planner: "mark 123 as Fuel" → the right CategorizeCommand; executor (with --yes) applies it; without confirmation, nothing is written.
  • A drafted command referencing an unknown id is rejected at execution (the deterministic layer validates, not the model).
  • Suite stays offline (fake planner; no Ollama).

ADR / AC flags

  • ask stays read-only — ADR-17 unchanged because the write lives in a separate verb. Confirm that framing before building.
  • No new financial AC; correctness is inherited from 026/027's deterministic executor.

Out of scope

  • Anything the model could decide unconfirmed. Multi-turn dialogue. Free-form rules ("always treat X as Y" — that's a merchants.yaml edit).

Definition of done (only if D0 = build)

  • NL phrase → drafted command → confirm → deterministic apply; no model-authored write ever lands unconfirmed; ask remains read-only.
  • Offline suite green; ruff/pyright/pytest clean.
