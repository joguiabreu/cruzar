Plan 016 — cruzar fetch (Gmail ingestion)

Slice goal. Add the last unbuilt command, cruzar fetch: poll Gmail, apply the per-account sender + subject allowlist from sources.yaml, and download PDF attachments into data/inbox/<account_match>/ — where cruzar process then ingests them (SPEC §Inputs, ADR-4/9/10-path). This is the email ingestion path that converges with the manual path at data/inbox/.

Two things that make this slice different from 001–015:
  • Not AC-gated. The acceptance harness (AC1–AC22) is already complete; no numbered AC tests fetch. "Done" here = fake-injected offline tests + ruff/pyright/pytest green + a real-run gate (you run it against your real mailbox). It also exercises ADR-9 (secrets in Keychain), which AC5 guards.
  • New dependencies + a Google Cloud prerequisite. Needs keyring and the Google API client, and you must create a Google Cloud OAuth client (enable the Gmail API, download the client JSON) before the real run — I can't do that part.

Decisions needing your sign-off

D1 — libraries

Add google-api-python-client + google-auth-oauthlib (official Gmail client & OAuth flow) and keyring (Keychain). Recommended over hand-rolled HTTP/OAuth — token refresh, retries, and the consent flow are fiddly and the official libs are the well-trodden path. Imports kept local to the fetch module so the rest of the app (and the offline suite) never loads them.

D2 — auth & secret storage (ADR-9)

OAuth installed-app flow. The refresh token (the durable secret) is stored in the macOS Keychain via keyring (service cruzar-gmail), never on disk; access tokens stay in memory. First run (cruzar fetch --login) opens a browser for consent and persists the refresh token; later runs load it and refresh silently.

The OAuth client config (client_id/secret JSON from Google Cloud) is user-provided, gitignored config at a path set in cruzar.yaml (gmail.client_secret_path). For an installed app the client secret isn't a true credential (it can't act without the user's consent), so treating it as gitignored config — while the actual tokens live only in Keychain — satisfies ADR-9 / AC5. Recommended; flag if you'd rather the client JSON also go through Keychain.

D3 — what fetch does (scope boundary)

For each source_type: email account: build a Gmail query from its sender_match + subject_match (+ has:attachment filename:pdf), list matching messages, skip already-fetched ones, download each PDF attachment to data/inbox/<account_match>/, and record it. It does not run process — fetch and process are separate idempotent steps (ADR-4); you run process after. Rate-limit / transient HTTP errors → exponential backoff; a sustained failure aborts the run cleanly (nothing half-written). Recommended.

D4 — resume / dedup state

A new fetched_messages(message_id PK, account_match, fetched_at) table in the DB: a Gmail message is downloaded once; re-running resumes cheaply and idempotently (the fetch analogue of file-hash dedup). Recommended over a loose state file — SQLite is the source of truth (ADR-3) and this stays transactional. Adds one guarded db._migrate step (schema-parity safe). Alternative if you'd rather not touch the schema: a gitignored data/fetch_state.json. Say which.

D5 — config schema

AccountConfig gains optional sender_match / subject_match (email-only; default None) so AccountConfig(**entry) still accepts manual entries. fetch validates: an email account must have both; a manual account ignores them. These stay config-only (read from sources.yaml at fetch time), not a DB column. Recommended.

D6 — multi-account-in-one-email (known break)

Per-account queries route each attachment to the querying account's folder. If one message matches two accounts it downloads into both folders; process's file-hash dedup keeps the first and skips the rest — the documented v1 "can't split one email across two accounts" limitation (SPEC §Edge cases), flagged, not split. No new work; confirm acceptable.

Design

New module fetch.py

  • GmailClient Protocol — search(query) -> list[MessageRef] and attachments(message_id) -> list[(filename, bytes)] — so the network client is injectable and tests pass a fake (the offline-suite pattern from the LLM tier).
  • fetch(conn, config, inbox_dir, *, client) -> FetchSummary: per email account, query → skip known message_ids → download PDFs to the account folder → record in fetched_messages. Pure orchestration; no Google imports here.
  • Google specifics (OAuth, token refresh via Keychain, the real GmailClient) live in a thin builder google_gmail_client(client_secret_path) with local imports — mirrors how llm.ollama_* isolates instructor/openai.

CLI

  • cruzar fetch → build the real client (refresh token from Keychain) → run fetch. cruzar fetch --login → run the consent flow once and store the refresh token. Prints a summary (downloaded / skipped / per-account counts), filenames only — never message bodies (the no-real-values-in-logs invariant).

Schema / migration

  • schema.sql: fetched_messages table; db._migrate guarded CREATE TABLE IF NOT EXISTS is enough (new table, not an altered one) — parity test confirms an upgraded DB matches.

Tests (offline; fake GmailClient injected — no network, no Keychain)

  • test_fetch_routes_attachments_by_allowlist — two email accounts + a fake client returning canned messages with PDF bytes; assert each PDF lands in its account's inbox folder and a fetched_messages row is recorded.
  • test_fetch_is_idempotent — a second run with the same fake re-downloads nothing (message_ids already recorded); the resume cursor holds.
  • test_fetch_email_account_missing_match_errors — an email account without sender/subject is rejected loudly (fail-loud config validation).
  • test_fetch_query_built_from_allowlist — the query string includes sender, subject, and filename:pdf (spy on the fake's search).
  • A conftest-level guard so pipeline/other tests never construct the real client.

SPEC + README

  • SPEC: no ADR/AC change (fetch is already specified). Optionally note the fetched_messages resume table where the Gmail rate-limit edge is described.
  • README: document cruzar fetch + --login, the one-time Google Cloud OAuth setup, the gmail.client_secret_path config and email-account fields (sender_match/subject_match), and that tokens live only in the Keychain (ADR-9). Update the sources.yaml template with an email example.

Out of scope

  • Scheduling (launchd/cron) — documented as user-run; not built.
  • Splitting one email across multiple accounts (D6) — v1 limitation.
  • Non-PDF attachments, non-Gmail providers, label/folder management.
  • Auto-running process after fetch (separate step, ADR-4).
