Plan 026 — Manual categorization (cruzar categorize)

Slice goal. Fill the manual tier's missing write path (ADR-13: manual > rule > llm). Let me correct a single transaction's category by hand — for a one-off that should NOT become a committed brand rule (e.g. a counterparty-name transfer, a mis-read line). Deterministic CLI, no LLM, no network. This is Phase 1 of a 3-part roadmap (026 categories · 027 is-transfer override · 028 optional NL sugar).

Why a command, not "ask the LLM to classify it". cruzar ask is read-only by ADR-17 ("reader, not advisor"); ADR-1 says the model never decides financial truth. "Classify 12345 as Fuel" has zero ambiguity needing a model — the id is an exact token, the category is a closed vocabulary. Routing it through a 3B model risks it emitting 12354 and silently mis-assigning money, with no guard to catch it. So the mutation is a deterministic command; the model adds risk, not value. (NL drafting, done safely, is deferred to plan 028.)

The gap (and what's already built)

  • merchant_source already allows 'manual' (schema CHECK), and the rule + llm passes already skip manual rows (WHERE merchant_source != 'manual'), so a manual decision is durable and never clobbered. AC17 already asserts "manual rows are never modified by a run." The plumbing is reserved — only the write path is missing.
  • Wrinkle: category is not on the transaction — it lives on the merchant (transactions.merchant_id → merchants.category). So a per-transaction category override does not fit the shared-merchant model cleanly (see D2).

Decisions needing your sign-off

D1 — a deterministic CLI command

cruzar categorize <id...> --category <Name> [--clear]. No LLM, no network, read-only w.r.t. everything except the targeted rows. Recommended — preserves ADR-17/ADR-1 and is faster + safer than NL for exact ids and a closed vocabulary.

D2 — store the override as a transaction-level category (KEY fork)

Category lives on the shared merchants row, so "this one txn is Fuel" can't be expressed by re-pointing to a merchant without mutating every txn of that merchant. Two options:

OptionMechanismCost
A — new column (recommended) Add nullable transactions.manual_category (a guarded migration in db._migrate). Metrics read COALESCE(t.manual_category, m.category, 'Uncategorized'). One migration + a small change in the spend/income grouping SQL; semantically exact for a one-off.
B — per-txn merchant Create a dedicated merchant per overridden txn with the chosen category. Pollutes the merchants table; weird for analytics. Rejected.

Recommend A. It touches the schema (migration) and the grouping metrics — both flagged below. Setting a custom merchant name is out of scope here (category is the 80% need); a manual-merchant label can come later.

D3 — reference by id, identity is content_hash

You pass the transaction id (ergonomic, shown in the report — see D5). The override is written on that row; the row is content_hash-stable and ingestion is idempotent (AC3), so the override survives reprocessing for free. A separate re-appliable manual_overrides table (keyed by content_hash, re-applied each run like a rule) is deferred unless we find DB rebuilds actually happen.

D4 — reversible & idempotent

--clear reverts a row to merchant_source='none' (nulls manual_category) so the next run's rule/llm pass re-decides it. Re-running the same command is a no-op. The passes already skip manual rows, so an override needs no special protection beyond what AC17 guarantees.

D5 — surface the handle

You can't act on a transaction you can't name. Add the txn id (and the amount/date already shown) to the report's Needs Categorization section so a flagged line reads as an actionable target. Open sub-choice: raw id vs a short content_hash prefix — I lean raw id for typing ease.

Design / modules

  • schema.sql + db._migrate — add nullable transactions.manual_category via a guarded migration step (never CREATE-only; schema-parity test must stay green).
  • categorize.py — a set_manual(conn, ids, category) / clear_manual(conn, ids) that validates the category against the vocabulary, sets manual_category + merchant_source='manual', and refuses an unknown id or off-vocab category (fail loud).
  • metrics.py — spend/income grouping reads COALESCE(t.manual_category, m.category, 'Uncategorized') so a manual override shows in Spending-by-Category and the totals still reconcile.
  • cli.py — cruzar categorize subcommand (ids + --category/--clear); prints what changed.
  • report.py — id in the Needs-Categorization rows (D5).

Tests

  • Manual set links the category, sets source='manual'; a following rule/llm pass leaves it untouched (reuses the AC17 guarantee).
  • --clear reverts to none and the next pass re-decides.
  • Metrics: a manually-overridden txn lands in its category and Spending-by-Category still sums to Spent.
  • Unknown id / off-vocab category → clear error, no write.
  • Schema-parity + pipeline-smoke stay green (migration upgrades an old DB).

SPEC / README / ADR-AC flags

  • Implements ADR-13's manual tier — no ADR change. AC17 already covers the invariant; add a focused unit/AC-style test for the new write path.
  • Schema migration (the one invariant this touches) — guarded _migrate step + frozen-baseline parity.
  • SPEC: short note under categorization that manual is settable via cruzar categorize. README: document the command + that it survives reprocessing.

Out of scope (→ later plans)

  • Marking something a transfer / not-a-transfer → plan 027.
  • Natural-language "classify 123 as Fuel" → plan 028 (optional).
  • "All txns like this → category" — that's already a rule (merchants.yaml), not a manual override.
  • Setting a custom manual merchant name (only category here).

Definition of done

  • cruzar categorize <id> --category X sets a durable manual override that shows correctly in reports and survives process; --clear reverts.
  • Migration upgrades an existing DB; schema-parity + smoke green.
  • ruff + pyright + pytest clean; SPEC + README updated.
