Headline: this file is a cash ledger, not a portfolio report

data/inbox/degiro/Account.pdf is Degiro's Account statement — a chronological cash-account ledger (deposits, withdrawals, fees, interest, buys, internal sweeps) with a running Balance. It does not contain a positions/portfolio table (no per-security quantity/value). Consequences, which set expectations for a "broker account":

  • ✅ We can ingest the cash flows + closing cash balance — this fits the existing ParsedStatement model with account_type: brokerage. ADR-11's requirement that investment parsers emit closing_balance + cash-flow transactions is satisfied.
  • ⛔ We cannot populate holdings_snapshot (no positions here), so securities value is unknown. Net Worth (ADR-16 = cash balance + Σ holdings value) would see Degiro's cash balance but zero securities; Portfolio Δ (ADR-14) can't be computed for Degiro until positions exist.
  • To track Degiro holdings / portfolio value we'd need Degiro's separate Portfolio export (positions with quantity + value) — a future parser/slice. This slice deliberately defers it (analogous to Moey's deferred APLICAÇÕES section).

Findings (real file, digit/PII masked)

  • 4 pages, wide/landscape. Header: Date | Time | Value date | Product | ISIN | Description | FX | Change | Balance. Period line: Account statement from DD-MM-YYYY until DD-MM-YYYY (English). Account currency EUR.
  • Column geometry (token x0): Date 35 · Time 81 · Value date 110 · Product 163 · ISIN 329 · Description 403 · FX 603 · Change ≈636 (a EUR token + amount) · Balance ≈705 (a EUR token + amount). Single layout — fixed x-bands are fine (unlike Revolut).
  • Number format: space thousands, comma decimal, with a currency-code prefix — e.g. EUR 1 234,56, negatives EUR -1,23. (Distinct from Moey's dot-thousands and Revolut's € dot-decimal.) The space splits the integer into separate tokens (1, 234,56) → join then parse, like ActivoBank.
  • FX column: a rate appears for USD trades; informational — kept verbatim in description_raw, never used for amounts (ADR-1, ADR-5: no FX math in the parser).
  • Multi-line rows: buys wrap (Compra NN Vanguard … USD then Acc@<price> EUR (ISIN)); cash-account moves wrap (… flatexDEGIRO Bank then AG: <amt> EUR). Continuations (no date) merge into the preceding row (ADR-11).
  • Row vocabulary (56 date-led rows; counts):

Parsing strategy

  1. Cluster rows (reuse _common). Locate the table by the header row (Date … Change Balance); skip the per-page header + footer.
  2. Transaction row = leading DD-MM-YYYY:
    • date = first date token (Date column).
    • amount = the Change column value: join the numeric tokens in the Change band (≈650–700, after its EUR), space-thousands + comma-decimal → signed Decimal.
    • description_raw = all tokens from the Product column up to the Change column (x0 163…<636) joined — keeps Product + ISIN + Description + FX verbatim.
  3. Continuation (no date): merge tokens into the previous row's description_raw (one logical line; seq does not advance).
  4. closing_balance = the Balance value of the last transaction row (band x0 ≥ ~720).
  5. period from the English "from … until …" line; currency = "EUR".
  6. Any failure raises DegiroParseError — nothing partial.

A _degiro_decimal(tokens) helper joins space-split integer groups, swaps comma→dot, strips the currency code, and returns Decimal.

Decisions

D1 — Scope: cash ledger only; holdings deferred. CONFIRM. Ingest the Account statement as a brokerage cash statement (transactions + closing cash balance). Defer holdings/positions — they aren't in this document; populating holdings_snapshot (and thus Degiro's portfolio value / Net Worth / Portfolio Δ) needs Degiro's separate Portfolio export, a later slice. Are you OK ingesting cash-only now, with portfolio value deferred? (If you have/can export the Portfolio PDF, we'd scope a follow-up to read positions into holdings_snapshot.)
D2 — Degiro number format helper. Space-thousands, comma-decimal, currency-code prefix; integer split across tokens. Add a Degiro-local _degiro_decimal (the three parsers now use three different number formats, so a per-parser boundary helper is clearest; no shared abstraction yet). Agree?
D3 — The 10 Conta Caixa rows with no Change value. Levantamentos/Depósitos da sua Conta Caixa na flatexDEGIRO Bank are the flatex-side mirror of a Cash Sweep Transfer (which we already capture with its own Change); their amount is stated in prose (… AG: <amt> EUR) and they carry no Change in the DEGIRO ledger. Options:
  • (a) Skip them (recommended) — they have no effect on the DEGIRO cash balance (mirror of the sweep) and emitting them risks a confusing 0-amount / double-counted line. To be confirmed at implementation by checking the Balance column is unchanged across them.
  • (b) Emit with amount = 0 — keeps every physical row; adds noise.
  • (c) Parse the prose amount — risks double-counting the sweep and ambiguous sign.
Recommendation: (a), verified against the balance column during build (same evidence-driven approach used for Revolut's edge cases).
D4 — Brokerage wiring & deferred detection. account_type: brokerage. Brokerage transactions are already excluded from Spending Detail by the report's _CASH_TYPES=(checking,savings) filter — no report change, AC19 respected. Deferred (out of scope, flagged): treating flatex Deposit as a NetContrib contribution and the sweeps as internal transfers — that's the flows.yaml investment_flow_patterns / Portfolio Δ slice. Agree to defer?
No model / schema / persist / report changes in this slice. The cash ledger fits the existing ParsedStatement. The first change to those will come with the Portfolio/positions slice (writing holdings_snapshot — which will touch the parser output contract and persist; that slice will stop-and-ask per the ADR rule).
